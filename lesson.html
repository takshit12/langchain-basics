01 · What LangChain is

A toolbox, not a framework.

LangChain is a library of components for working with LLMs — models, prompts, tools, memory, retrievers. It is not an agent framework itself. It is the box of parts.

Models
A standard wrapper around any LLM provider.
Prompts
Reusable instructions with fill-in variables.
Tools
Functions the model can decide to call.
Memory
Conversation state, short- and long-term.
Retrievers
Fetch relevant documents for a question.
Parsers
Turn raw model text into structured data.

01 · The core problem it solves

A raw LLM call is text-in, text-out.

A model alone

  • Does not remember the last turn
  • Cannot look anything up — no live or private data
  • Cannot take action — no tools, no side effects
  • Only knows what it was trained on

LangChain adds

  • Memory — carry state across turns
  • Retrievers — pull in the right documents
  • Tools — let the model act on the world
  • Standard, swappable pieces for each

01 · The core idea

What is a chain?

A chain is just several small steps wired in a row, where each step's output becomes the next step's input. Instead of one giant LLM call, you do the work in ordered stages.

Think of it like

A recipe or an assembly line: chop → cook → plate. Each stage does one job and passes its result down the line. LangChain just lets the stages be LLM calls, lookups, tools, and formatters.

01 · The mental model

Pick parts. Wire them. Run once.

Working in LangChain is always the same three moves — no matter how big the app gets.

1 · Pick
Choose the components you need — a chat model, a prompt, maybe a tool or a retriever.
2 · Wire
Connect them into a chain, in the order the data should flow through them.
3 · Run
Call the chain once with .invoke() — LangChain runs each step and passes the data along.
The shift in thinking

You describe the pipeline — the parts and their order — and LangChain runs it for you. You are not hand-writing the plumbing; you are declaring the shape.

01 · What's in the box

The parts you chain together.

These are the main component families. Every one is a Runnable — one link you can drop into a chain. We go deeper on each in a moment; this is the map first.

Chat Models
The LLM call itself — one standard wrapper around any provider.
Prompt Templates
Reusable instructions with {blanks} filled at runtime.
Output Parsers
Turn the model's raw text into a string, JSON, or a typed object.
Tools
Functions the model can choose to call to take action.
Memory
Carry context across the turns of a conversation.
Retrievers
Fetch the right documents to ground an answer (the R in RAG).

01 · Composability

A chain is itself a component.

Here is the part that makes it scale: once you wire steps into a chain, that whole chain is also a Runnable — so you can drop it inside a bigger chain, make it a LangGraph node, or hand it to an agent. Small pieces nest into big ones.

  • A single step can be one model call or an entire sub-chain — the outside can't tell.
  • That is why the same building blocks power a one-line demo and a full agent.
You have already seen this

Every LangGraph node you built wrapped a small LangChain chain — a model or a tool call. Same parts, one level up. The abstraction changed; the pieces did not.

01 · A request's journey

One question, step by step.

Watch a single request flow through a simple prompt | model | parser chain — and notice how the data changes shape at each hop.

  • You pass in a small dict — {"topic": "Runnables"}.
  • Prompt fills the template → a ready-to-send message.
  • Model reads the message → a raw AI message (text + metadata).
  • Parser strips it down → a clean string you can use.
Why it just works

Each step's output is a valid input to the next because they all share one interface. That interface has a name — the Runnable — and it is the very next slide.

01 · The standard interface

One shape: the Runnable.

Every component — a model, a prompt, a retriever, a parser — follows the same Runnable pattern. Each one takes an input, does its job, returns an output.

  • Because they all share one interface, they all speak the same language.
  • That is why they chain — the output of one is a valid input to the next.
  • You compose them with the LCEL pipe: prompt | model | parser.
The one idea to carry all day

If it is a Runnable, it pipes. LangGraph nodes, LCEL chains, and agents are all the same pieces — recombined at a different altitude.

01 · Why the standard interface matters

Swap any part without rewiring the rest.

Because the shape is fixed, the provider behind a step is just a detail. OpenAI, Anthropic, Gemini — swap the model chip, the chain around it does not change.

Swap the model
GPT → Claude → Gemini. The prompt and parser stay put.
Swap the retriever
FAISS today, a hosted DB tomorrow. Same slot.
Swap the parser
Plain string → JSON → typed object. No rewrite.
Proven in this repo

In langchain_fundamentals.py, with no API key the model step is replaced by a canned RunnableLambda — and the exact same prompt | model | parser chain still runs. Swap-ability, demonstrated live.

01 · Building block — Chat Models

Chat Models

A standard wrapper around any LLM provider. One consistent interface — .invoke() a message, get a message back — no matter who serves the model.

  • ChatOpenAI, ChatAnthropic, ChatGoogleGenerativeAI — same shape.
  • It is a Runnable, so it drops straight into any | chain.
Tie-back

Every "LLM node" you built in LangGraph was, underneath, one of these chat-model calls.

01 · Building block — Prompt Templates

Prompt Templates

Reusable instructions with variables filled in at runtime. Write the wording once; drop in different values on every call.

ChatPromptTemplate.from_template(
  "You are a concise teaching assistant. In ONE short "
  "sentence, explain: {topic}"
)
Tie-back

The system prompts you hard-coded into LangGraph nodes are exactly this — just without the reusable {variable} slot.

01 · Building block — Output Parsers

Output Parsers

Turn raw LLM text into structured data — a clean string, a JSON object, a list, a typed object — so the next step gets something reliable instead of free-form prose.

StrOutputParser
Pulls the plain text out of the model's message. The simplest one.
JsonOutputParser
Parses the reply into a JSON object your code can read.
Pydantic parser
Validates the reply against a typed schema.
Why it matters

A parser is the last chip in the pipe — it makes the model's answer usable by the rest of your program.

01 · Building block — Tools

Tools

Functions the LLM can decide to call. This is the building block behind every "agent taking action" you have ever seen.

@tool
def word_counter(text: str) -> int:
    """Count how many words are in the given text."""
    return len(text.split())

The decorator auto-derives the tool's name, description, and typed argument schema — exactly what gets advertised to the model. The model reads that and chooses whether to call it.

01 · Building block — Memory

Memory

Short-term

  • Context within one conversation
  • The running message history on a turn

Long-term

  • Persisted across sessions
  • Recall who a user is next week
Tie-back — you already used this

This is the same concept as the state and checkpointing you wired in LangGraph. Memory is just that idea, named as a first-class LangChain building block.

01 · Building block — Middleware

Middleware

Hooks at points in the agent loop — before or after a model call, before or after a tool call. Standard slots to inject control without rewriting the loop.

Tie-back

Before middleware, this kind of control meant hand-building custom logic inside a LangGraph node. Middleware is LangChain's standard way to give you that same control.

01 · Middleware — named built-ins

Four you can reach for today.

Human-in-the-loop
Pause for a person to approve a tool call before it runs.
Summarization
Compress history to prevent context overflow on long runs.
Automatic retries
Re-try a failed model call with backoff instead of crashing.
Content moderation
Screen inputs or outputs against a safety policy.

This repo uses ModelRetryMiddleware — the "automatic retries" example, a real built-in.

01 · Building block — create_agent

create_agent — model + tools, assembled.

The fast, standard way to build a working agent loop. What you hand-built in LangGraph, in a few lines — and it compiles to a LangGraph graph internally.

02 · How it all stacks up

Four layers. Same foundation.

Read top to bottom: more abstracted at the top, closer to the raw model at the bottom.

02 · The orchestration layer

LangGraph = the runtime.

LangGraph takes LangChain components — models, tools — and arranges them into a graph: nodes, edges, state, checkpoints.

  • Every "LLM node" you built sat on top of a LangChain chat-model call.
  • Every "tool node" was a LangChain tool.
  • The graph decides order and control flow; the chips inside are pure LangChain.
Same pieces, new altitude

LangGraph does not replace the components — it choreographs them.

02 · One layer further up

Deep Agents = assembled architecture.

Deep Agents are built the same way, one layer higher — agent patterns like planning, sub-agents, and file systems, assembled from LangGraph running LangChain components.

Planning
Break a goal into steps before acting.
Sub-agents
Delegate a slice of work to a focused agent.
File systems
A scratchpad the agent can read and write.

Peel any of these back and you find a LangGraph graph — and inside it, LangChain model, tool, and prompt calls.

02 · The point to land

It is always LangChain calls underneath.

Land this sentence

Whether you write raw LangChain, hand-build a LangGraph, or use a pre-built Deep Agent, you are always ultimately making LangChain model / tool / prompt calls.

The abstraction level changes. The foundation does not.

02 · One action, traced through the stack

An agent calls a tool — top to bottom.

  • Deep Agent — "I should check the word length." Decides to use a tool.
  • LangGraph — routes the run from the model node to the tools node.
  • LangChain — the @tool function runs; its result becomes a tool message.
  • Chat Model — reads the tool result and writes the final answer.

Same event, four names for what is happening — one at each layer.

03 · The problem RAG solves

A model only knows its training data.

It cannot answer about your private or internal documents, or anything that happened after it was trained. Ask it about your company's policy and it will guess.

Out of reach for a plain model

  • Your internal wiki and support docs
  • Last week's release notes
  • Contracts, tickets, records in your database

RAG makes these answerable

  • Fetch the relevant text first
  • Hand it to the model as context
  • Let it answer from what it was given

03 · The one analogy

Closed-book vs open-book exam.

Closed book — plain LLM

  • Answers from memory alone
  • Confident even when it is wrong
  • No way to check the specifics

Open book — RAG

  • Looks up the relevant page first
  • Answers from what it just read
  • Grounded, and you can see the source

RAG hands the model the right page, open, before it answers.

03 · Unpacking the acronym

R · A · G

R — Retrieve
Find the chunks of text most relevant to the question.
A — Augment
Insert those chunks into the prompt as context.
G — Generate
The model answers using that context instead of guessing.
The shift

From "what does the model remember?" to "what did we just give it?"

03 · The full RAG pipeline

Two flows, one join.

Indexing runs once up front. Every question flows through the bottom two lanes.

03 · Pipeline stage 1 — Loading

Loading

Pull the raw document in — a PDF, a webpage, a database record — and turn it into a LangChain Document object the pipeline can work with.

[1/4] Loading document: docs/kestrel_home_faq.md
      loaded 1 document(s), 5058 characters total

Different sources, one common Document shape — so the rest of the pipe does not care where the text came from.

03 · Pipeline stage 2 — Splitting

Splitting into chunks

Cut the document into smaller pieces. Smaller chunks retrieve more precisely — and a model cannot process a huge document in one shot anyway.

[2/4] Splitting into chunks (chunk_size=500, chunk_overlap=50)
      produced 16 chunks

A small overlap keeps a sentence that straddles a boundary from being cut in half and losing its meaning. The Kestrel doc splits into 16 chunks.

03 · Pipeline stage 3 — Embedding

Text becomes geometry.

Each chunk of text becomes a vector of numbers. Similar meanings land at similar vectors — so "restocking fee" sits near "returns policy," far from "wifi hours."

all-MiniLM-L6-v2 maps every chunk to a 384-number point. Runs locally, no API key.

03 · Pipeline stage 4 — Vector DB & similarity

Closeness in meaning, not keywords.

A vector database stores those vectors and, given a new query vector, finds the ones closest to it.

Similarity search = finding chunks whose meaning is nearest the question's meaning — not exact keyword matching. Ask "late-return charge" and it still finds "restocking fee."

03 · Which vector database?

Same idea, different scale.

FAISS
Lightweight and local. Just a file on disk — no server, no account, no key. Perfect for a demo or a single machine. This repo uses FAISS.
Pinecone · Chroma
Hosted, larger-scale. Managed services that hold far more vectors and serve many users. Same concept, run as infrastructure.
The mental model

A vector DB indexes by closeness in meaning instead of exact match — not fundamentally different from a regular database, just a different notion of "matching."

03 · Pipeline stage 5 — Retrieval

Retrieval

The question is embedded the same way the chunks were. The database returns the closest-matching chunks — the ones whose meaning is nearest the question.

  • Same embedding model for indexing and querying — so both live in the same space.
  • Usually the top-k chunks (here, k=3).
  • A Retriever is itself a Runnable — so it drops into the pipe.
This is the "R" in RAG

No model has answered yet — we have only found the right pages.

03 · Pipeline stage 6 — Augmenting

Augmenting the prompt

The retrieved chunks get inserted into the prompt as context, right alongside the original question.

Answer the question using ONLY the context below.
If the context does not contain the answer, say you
don't have that information -- do not guess.

Context:
{retrieved chunks go here}

Question: {the user's question}

"Answer only from this context" is what keeps a grounded answer from drifting back into guessing.

03 · Pipeline stage 7 — Generation

Generation

The model answers using that context instead of guessing. Same chat model as always — it just has the right page open in front of it now.

Grounded
The answer traces back to a retrieved chunk.
Honest
If the context lacks it, the model can say so.
Current
Swap the document, and the answers update.

Retrieve → Augment → Generate. The last step is the only one the model itself does.

03 · The tie-back

RAG is the same pipe — with a retriever spliced in.

Not a new paradigm

04 · Live demo — run of show

Four steps, watch the answer flip.

1 · Ask plain
Ask a plain LLM about the private Kestrel Home doc it has never seen.
2 · Ask with RAG
Run the same question through the full retrieve → augment → generate pipeline.
3 · Show chunks

Question: "What is Kestrel Home's restocking fee on returns outside the 45-day window?"

04 · The "before"

Plain LLM — confidently wrong.

========================================================================
PLAIN LLM  (no document access -- this is the 'before')
========================================================================
"Kestrel Home charges a 15% restocking fee on returns that
 are made outside their 45-day return window."
^ A real plain-LLM answer. Specific and confident -- and
  wrong: the model never saw the doc, so it guessed the fee.
This is the moment that sells RAG

It is not vague — it is specific and wrong. The 15% fee is invented; the document actually says 12%.

04 · The "after"

RAG — grounded and right.

[retrieved 3 chunks from 'faiss_index/']
chunk[0]: - A **12% restocking fee** applies to opened returns outside
  the 45-day window but within the 26-month warranty period,
  UNLESS the return reason is a confirmed hardware defect (in
  which case the restocking fee is waived entirely).

[grounded answer]
Kestrel Home's restocking fee on returns outside the 45-day
  window is 12%, unless the return is a confirmed hardware
  defect -- in which case it is waived entirely.

Same question, same model — only RAG saw the document, so only RAG got 12% right. Index facts: the Kestrel doc → 16 chunks; the swap doc → 5 chunks.
